# Pain Points & Real Constraints

These are not "bad code" — they're **real constraints** that any solution must work within.

## 1. Qwen's Single-Session Limit

```
Bad_Request: The chat is in progress!
```

Qwen's backend only allows **one active generation per session**. This is the single biggest constraint. Consequences:
- Global Mutex on chat completions (serializes all requests)
- Retry logic with exponential backoff
- Session parent-message tracking to maintain conversation continuity
- No true concurrent request support

**Workaround Potential**: Could multiple sessions be opened via different browser profiles? The code shares one `qwen_profile/` context. Multiple profiles = multiple sessions = parallel generations.

## 2. Anti-Bot Tokens (bx-ua, bx-umidtoken, bx-v)

These are dynamically generated by Qwen's JavaScript. You can't:
- Hardcode them (they expire/change)
- Get them from a simple HTTP request
- Predict them algorithmically

Only approach: **run a real browser and intercept them**.

## 4. Browser Dependency

The proxy **requires**:
- Playwright installed (`npx playwright install`)
- A browser engine (chromium, firefox, webkit, chrome, edge)
- A valid login session (manual or auto-login)
- The `qwen_profile/` directory for persistent context

This makes it:
- Heavy (~500MB+ with browser)
- Fragile to Qwen UI changes
- Slow to start (browser launch + session check)
- Non-trivial to containerize (Docker image is 1.5GB+)

## 5. Cumulative vs Incremental Streaming

Qwen's stream is **inconsistent**:
- Sometimes returns the **full cumulative text** each chunk
- Sometimes returns **only the delta** (new text since last chunk)

The `getIncrementalDelta()` function uses a heuristic (common prefix length ≥ 4 chars) to detect which mode it's in. This can produce:
- **Correct deltas** (~90% of cases)
- **Double characters** when heuristic mis-fires on repetitive text (like `import {`)
- **Missing characters** when the stream switches modes mid-generation

## 6. Tool Calling as a Prompt Hack

Qwen doesn't have native function calling. The entire tool system is a **prompt injection**: tool descriptions are serialized into the system prompt, and the model is instructed to output `<tool_call>` tags. This means:
- Every request carries the full tool schema in the prompt (wastes tokens)
- Qwen can "forget" to use the tag format
- Different Qwen models may format tool calls differently
- No streaming-native tool call support (tool calls appear mid-text, not as structured data)

## 7. RobustParseJSON Complexity (198 lines)

The JSON parser handles real LLM output pathology:
- Unquoted property names: `{name: "test"}`
- Double keys: `{"name": "name": "test"}`
- Trailing garbage after JSON: `{"a": 1})`
- Control characters in strings: literal `\n` instead of `\\n`
- Windows paths: `C:\Users\`
- Truncated JSON: `{"a": 1`
- Incomplete escapes: `\\g` (invalid escape)

Each fix was added because a real Qwen response broke without it.

## 8. Global Mutable State

| State | Purpose | Why Global |
|---|---|---|
| `sessionStates` | Parent message ID tracking | Survive hot-reloads |
| `cachedQwenHeaders` | Auth token cache | Avoid browser dance every request |
| `currentHeaders` | Most recent bx headers | Quick lookup |
| `activePage` | Current browser page | All browser operations |
| `cachedModels` | Model list | Avoid re-fetching |
| `nativeToolsDisabled` | Flag for one-time setup | Prevent redundant API calls |

All module-level. Difficult to test, impossible to parallelize. But wrapping in DI would add complexity without user-facing benefit for current use cases.
